## 1. 最上位原則 — 長期思考と信頼

全ての判断の土台。迷ったらここに戻ること。

### 1-1. 長期的に考える

- 短期的な売上より、ユーザー体験を優先する。ダークパターンで課金を迫るより、価値を先に届ける。
- 技術的負債を作らない。急いでハックするより、正しい設計で少し遅く出す方を選ぶ。
- 「今日の楽」より「半年後の自分が感謝するコード」を書く。

### 1-2. ユーザーからの信頼を第一にする

- ユーザーのプライバシーとデータを守ることは絶対条件。
- 約束したことは必ず実行する（無料トライアルの期間、機能の提供範囲など）。
- 透明性を持つ。課金条件、データの扱い、自動更新の条件を隠さない。

## 2. テスト駆動開発（TDD）— 最も重要な開発ルール

**コードを1行書く前にテストを書く。例外なし。**

### TDD サイクル（和田卓人 / Kent Beck 方式）

```
Red   → 失敗するテストを1つ書く
Green → そのテストだけを通す最小限のコードを書く
Refactor → テストが通ったままコードを整理する
```

### TDD ルール

1. 失敗するテストを書くまで、プロダクションコードを書いてはならない。
2. 失敗するテストは1つだけ書く。コンパイルエラーも失敗とみなす。
3. 失敗するテストを通すために必要最小限のプロダクションコードだけを書く。
4. リファクタリングはテストが全て通っている状態でのみ行う。
5. テストは仕様書である。テストを読めば何が実装されているか分かるように書く。

### テストの品質基準

- 1テスト = 1アサーション（原則）
- テスト名は日本語OK。「〜のとき〜になる」形式で何を検証しているか明確に。
- 境界値・異常系・エッジケースを必ず含める。
- モックは最小限。実際の動作に近いテストを優先する。

## 3. 開発ワークフロー

### 3-1. 計画を立ててから実装する

以下の順序を厳守すること:

```
① Plan Mode で計画を立てる（コードには触れない）
  - 既存コードを読む
  - 影響範囲を特定する
  - テスト戦略を決める
  - 実装方針を決める
② 計画を hiro に提示して承認を得る
③ 承認後、TDD サイクルで実装を開始する
```

**計画なしにコードを書き始めることを禁止する。**

### 3-2. リサーチは最新情報を使う

- 公式ドキュメント、最新の手順書を必ず読み込んでから計画を立てる。
- 古い情報や記憶に頼らない。APIの仕様、ライブラリのバージョン、プラットフォームのガイドラインは必ず最新を確認する。
- 特に App Store Review Guidelines, Google Play Policy は最新版を読むこと。

### 3-3. 小さく作って検証する

- PR は小さく。1つのPRで1つの機能。
- 完璧を目指さない。動くものを出して、フィードバックで磨く。
- ただし「動く」の定義にはテストが通ることを含む。

## 4. コード品質の基準

### 単一責任の原則（SRP）— 全ファイル・全関数に適用

**1ファイル = 1つの責任。1関数 = 1つの仕事。例外なし。**

この原則の目的は3つ:
- **可読性**: ファイルを開いた瞬間に「これは何をするものか」が分かる
- **保守性**: バグが起きたとき、修正すべきファイルが1つに絞れる
- **変更容易性**: ある機能を変えたとき、関係ないファイルが壊れない

具体的なルール:

1. **1ファイルに長いコードを書かない。** 目安として、1ファイル150行を超えたら分割を検討する。
2. **コンポーネントファイルにビジネスロジックを書かない。** UIとロジックを分離する。
   - `components/` → 表示だけ（props を受け取って描画する）
   - `lib/` or `hooks/` → ロジック（計算、API呼び出し、状態管理）
   - `constants/` → 定数・設定値
   - `utils/` → 汎用ユーティリティ（日付整形、バリデーション等）
3. **「このファイルは何をするか」を一言で説明できないなら、責任が多すぎる。** 分割する。
4. **God Object / God Component を作らない。** 1つのファイルが多くのことを知りすぎていたら設計を見直す。
5. **関数は1つの引数群から1つの結果を返す。** 副作用がある関数と純粋関数を明確に分ける。

### 全プロジェクト共通

- TypeScript strict mode（any 禁止）
- 関数型アプローチ（副作用の分離、純粋関数の優先）
- 命名は意図を表す（何をするかではなく、なぜ存在するか）
- コメントは「なぜ」を書く。「何を」はコードで表現する

### React Native / Expo プロジェクト

- Expo Router の convention に従う
- ファイル構成: app/ (ルーティング), components/ (UI), lib/ (ロジック), constants/ (定数)
- テスト: Jest + @testing-library/react-native

### Flutter プロジェクト

- Riverpod で状態管理
- freezed でイミュータブルモデル
- テスト: flutter_test + mocktail

### Next.js プロジェクト

- App Router 使用
- Server Components 優先
- テスト: Vitest + Testing Library

## 5. セキュリティ — 絶対に守ること

- `.env`, `credentials.json`, API キー、トークンを git にコミットしない。
- ユーザーデータを外部サーバーに送信しない（明示的な同意がない限り）。
- API キーの取得・貼り付けは必ず hiro が行う。エージェントが勝手に発行・入力しない。
- 機密情報は `~/.config/` 配下に保存し、プロジェクトディレクトリ内に置かない。

## 6. 意思決定の優先順位

迷ったとき、上から順に優先すること:

```
1. ユーザーの安全・プライバシー
2. ストア審査の通過（App Store / Google Play）
3. ユーザーからの信頼（透明性・誠実さ）
4. ユーザー体験の質
5. テストカバレッジ・コード品質
6. 開発速度
```

## 7. ビジネス指標 — 意識すること

### 最重要KPI: LTV（顧客生涯価値）とコンバージョン率

全ての機能設計・UI改善・施策は、最終的に以下の指標を改善するために行う:

- **コンバージョン率（CVR）**: 無料ユーザー → 有料ユーザーへの転換率
- **LTV（Lifetime Value）**: 1ユーザーが生涯で支払う合計金額
- **継続率（リテンション）**: 有料ユーザーが解約せず継続する割合

### ただし信頼を損なう手法は使わない

LTV・CVR を改善する手法は「セクション1: 長期思考と信頼」に反しないものに限る。具体的に:

- ユーザーを騙して課金させるダークパターンは禁止
- 解約を意図的に困難にする設計は禁止
- 虚偽の緊急性（嘘のカウントダウン、架空の在庫制限など）は禁止
- 価値を先に体験させ、納得の上で課金してもらう設計を追求する
